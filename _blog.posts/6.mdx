## 基礎的な<span>Git</span>用語

#### はじめに

Git の基本的な用語についてまとめてみました。 <br />
これなんていうっけ?って時に閲覧してみてください。

#### 基本的な Git 用語

<span>ー</span> リポジトリ <span>ー</span>

<span>リポジトリ</span> とは、プロジェクトの全履歴を含むデータベースのようなもので、
Git を使ってファイルやディレクトリの変更履歴を管理する場所です。 <br />
つまりリポジトリという場所に変更を加える予定のあるディレクトリやファイルを置いて管理するといったイメージです。

<br />
<br />

<span>ー</span> コミット <span>ー</span>

リポジトリ内で既存のソースコードを変更・削除したり、新たにディレクトリやファイルを追加したりします。
ある程度これらの作業が終わって一区切りした時に、一連の作業結果を記録したいと思いますよね。 <br />
その記録する作業を<span>コミット</span>と言います。コミットを繰り返し行うことで、
リポジトリにコミットしたときの履歴が溜まっていきます。またコミットには、
作業の内容を説明するメッセージを添えることができ、これが後で変更内容を追跡する際に役立ちます。

<br />
<br />

<span>ー</span> ブランチ <span>ー</span>

履歴を枝分かれさせて、分岐させるものを<span>ブランチ</span>と言います。 <br />
例えばベースとなるブランチがあったとして、変更を加えるためにベースを元に新しいブランチを作成し、
そこに変更内容を追加していきます。この新しいブランチをいくら変更しても、ベースになっているリポジトリは、
変化しないということです。またベースとなっているリポジトリは、<span>main ブランチ</span>
/<span> master ブランチ</span>という名前に、一般的にはなっています。

<br />
<br />

<span>ー</span> マージ <span>ー</span>

変更を加えたブランチを main ブランチに取り込むことを<span>マージ</span>と言います。 <br />
変更内容を main ブランチに追加するという作業はチーム開発をする上では特に重要で、
お互いに違う機能を作成しても main ブランチにマージすると決めておけば、同時並行で違う作業を実行でき、
追加できる仕組みが成り立ちます。 <br />
ちなみに、もしチームメンバーが同じファイルに、異なった変更を加えてマージしてしまったときは、
コンフリクトというエラーメッセージが来るので、そこは気をつけてください。
